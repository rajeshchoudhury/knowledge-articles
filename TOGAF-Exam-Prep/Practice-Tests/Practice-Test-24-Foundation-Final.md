## Answer Key

### Question 1
**Correct Answer:** a) Phase E identifies the major implementation projects and groups them into Transition Architectures; Phase F creates the detailed Implementation and Migration Plan with prioritization and sequencing
**Explanation:** Phase E (Opportunities and Solutions) focuses on identifying the major work packages, grouping them into meaningful groups, evaluating candidate solutions, and defining Transition Architectures—the intermediate stable states between baseline and target. Phase F (Migration Planning) takes those identified opportunities and creates the detailed, finalized Implementation and Migration Plan with prioritized sequencing, cost-benefit analysis, and risk assessment. Think of Phase E as "what to do" and Phase F as "in what order and how to get there."

### Question 2
**Correct Answer:** b) ABBs define architecture requirements and guide the specification of SBBs; SBBs represent candidate solution components that implement ABBs
**Explanation:** Architecture Building Blocks (ABBs) are architecture-level components that capture what is needed (requirements, specifications, functional descriptions). They are technology-neutral and describe desired capability. Solution Building Blocks (SBBs) are solution-level components that represent how the requirements are met—specific products, technologies, and implementations. The ABB-to-SBB mapping ensures traceability from architecture requirements through to solution design.

### Question 3
**Correct Answer:** b) A Viewpoint defines the perspective from which a View is taken; a View is the representation of an architecture from that specific perspective for a particular stakeholder
**Explanation:** In TOGAF, a Viewpoint is a reusable specification of the conventions for constructing and using a View—it defines the perspective, concerns addressed, audience, and modeling techniques. A View is an instance of a Viewpoint applied to a specific architecture—the actual representation created for particular stakeholders. For example, "Security Viewpoint" is the template; the "Order Management System Security View" is the specific representation for that system's security stakeholders.

### Question 4
**Correct Answer:** a) The Architecture Vision is a high-level aspirational description of the Target Architecture; the Statement of Architecture Work is a project document defining the scope, approach, and governance of the architecture effort
**Explanation:** The Architecture Vision describes what the target architecture will look like at a high level—the aspirational end state with key views and business value articulation. The Statement of Architecture Work (SoAW) is more operational—it defines the project parameters for the architecture engagement: scope, constraints, schedule, resources, governance approach, and approval. The SoAW is the "project charter" for the architecture work; the Vision is the "what we're building toward."

### Question 5
**Correct Answer:** b) Business Principles define the organization's values and strategic direction; Architecture Principles are derived from Business Principles and provide specific guidance for architecture decisions and IT governance
**Explanation:** TOGAF establishes a clear hierarchy: Business Principles express the organization's core values and strategic direction (e.g., "Customer data is an enterprise asset"). Architecture Principles are derived from and must align with Business Principles, providing specific guidance for architecture decisions (e.g., "All customer data must be managed through the master data platform"). Architecture Principles serve as a bridge between business intent and technical implementation. They do not override Business Principles but rather implement them.

### Question 6
**Correct Answer:** b) Compliant means the implementation includes all features of the specification plus additional non-standard features; Conformant means the implementation conforms to the specification without including additional non-standard features beyond it
**Explanation:** This is one of the most commonly confused distinctions in TOGAF. "Compliant" means the implementation covers all specified features but also includes features that are NOT in the specification—it goes beyond the spec. "Conformant" means the implementation includes all specified features and only those features—it matches the spec without extras. "Fully Conformant" goes further, meaning the implementation can serve as a direct substitute for the specification. The ordering from least to most aligned: Consistent < Compliant < Conformant < Fully Conformant.

### Question 7
**Correct Answer:** a) An artifact is a specific architectural work product (catalog, matrix, or diagram) that describes an aspect of the architecture; a deliverable is a contractually specified, formally reviewed and approved work product that may contain multiple artifacts
**Explanation:** TOGAF makes a clear distinction: artifacts are the building blocks of documentation—catalogs listing entities, matrices showing relationships, and diagrams providing visual representations. Deliverables are the formal work products that undergo review and sign-off—they typically contain and package multiple artifacts. For example, the Architecture Definition Document (a deliverable) contains the Application Portfolio Catalog (an artifact/catalog), the Application-Function Matrix (an artifact/matrix), and the Application Communication Diagram (an artifact/diagram).

### Question 8
**Correct Answer:** b) A deliverable is a work product of the ADM process (a document or specification); a building block is a potentially reusable component of business, IT, or architectural capability that can be combined with other building blocks to deliver architectures and solutions
**Explanation:** Deliverables and building blocks serve fundamentally different purposes. Deliverables are documentation work products of the architecture process—they describe and specify. Building blocks are reusable components (functional or implementation-level) that make up the architecture itself—they are what is being described. The Architecture Definition Document (deliverable) describes the architecture; an "Customer Identity Management" ABB (building block) is a component of that architecture.

### Question 9
**Correct Answer:** b) Strategic Architecture provides a long-term summary view of the entire enterprise; Segment Architecture provides detail for a specific business segment or functional area; Capability Architecture provides detail for a specific change initiative or project
**Explanation:** These three levels form the TOGAF Architecture Landscape: Strategic Architecture (broadest scope, lowest detail, longest time horizon—typically 3-10 years, covering the entire enterprise), Segment Architecture (medium scope and detail, typically for a business division or cross-cutting function), and Capability Architecture (narrowest scope, highest detail, specific to a project or change initiative). They are not sequential ADM phases but rather concurrent levels of architectural detail.

### Question 10
**Correct Answer:** a) Phase B addresses business strategy, governance, organization, and key business processes; Phase C addresses Data Architecture and Application Architecture that support the business architecture
**Explanation:** Phase B (Business Architecture) develops the business architecture that describes how the enterprise needs to operate to achieve its goals—covering business processes, organizational structure, roles, functions, and business services. Phase C (Information Systems Architectures) then develops the Data Architecture and Application Architecture that support the business operations defined in Phase B. Phase C has two sub-phases: Data Architecture (the structure of logical and physical data assets) and Application Architecture (the applications that manage the data and support business functions).

### Question 11
**Correct Answer:** a) The ADD describes the Target Architecture (what the architecture looks like); the ARS specifies the requirements that the architecture must satisfy and that implementation projects must meet (what the architecture must do)
**Explanation:** The Architecture Definition Document (ADD) is the architectural blueprint—it contains the baseline and target architecture descriptions across all four domains (Business, Data, Application, Technology), including views, diagrams, and building blocks. The Architecture Requirements Specification (ARS) complements it by specifying the measurable requirements, constraints, and success criteria that the architecture and subsequent implementations must satisfy. Think of ADD as "the design" and ARS as "the acceptance criteria."

### Question 12
**Correct Answer:** b) The Statement of Architecture Work is a project-initiation document that defines the architecture engagement scope; the Architecture Contract governs the implementation, ensuring that development partners deliver architecture-conformant solutions
**Explanation:** The Statement of Architecture Work (SoAW) is created at the start of an ADM cycle (Phase A) and governs the architecture development effort itself—scope, timeline, resources, governance. Architecture Contracts are created during Phase G (Implementation Governance) and govern the implementation projects—they ensure that solution builders deliver implementations that conform to the target architecture. The SoAW controls architecture work; Architecture Contracts control implementation work.

### Question 13
**Correct Answer:** b) The TRM provides a generic taxonomy and visual representation of the Application Platform and its services; the III-RM focuses on the components and services needed to enable Boundaryless Information Flow across the integrated information infrastructure
**Explanation:** The TRM (Technical Reference Model) is a foundational reference model providing a generic taxonomy of platform services—it classifies the services that an Application Platform should provide to applications (data management, user interface, security, etc.). The III-RM (Integrated Information Infrastructure Reference Model) is a more specific reference model focused on the infrastructure needed for Boundaryless Information Flow—the vision of seamless information sharing across organizational and system boundaries. The TRM is about platform services taxonomy; the III-RM is about information integration infrastructure.

### Question 14
**Correct Answer:** b) A concern is an interest or issue important to a stakeholder that may be broad or qualitative; a requirement is a specific, quantitative statement derived from concerns that defines what the architecture or implementation must satisfy
**Explanation:** Concerns and requirements are related but distinct. A concern is a stakeholder's interest or worry—it may be qualitative, broad, or even emotional ("I'm worried about system performance" or "We need to reduce operational risk"). Requirements are formal, specific, measurable statements derived from analyzing concerns ("The system must process 10,000 transactions per second" or "Mean time to recovery must be less than 4 hours"). The ADM process transforms stakeholder concerns into architecture requirements.

### Question 15
**Correct Answer:** a) Phase G ensures that implementation projects conform to the target architecture through compliance reviews; Phase H manages changes to the architecture itself in response to new drivers, technology changes, or implementation feedback
**Explanation:** Phase G (Implementation Governance) is about ensuring implementations follow the architecture—conducting compliance reviews, managing Architecture Contracts, and overseeing that what gets built matches what was designed. Phase H (Architecture Change Management) is about managing changes to the architecture itself—when new business drivers emerge, technology changes occur, or implementation experience reveals issues with the architecture, Phase H provides the process for updating the architecture baseline. Phase G protects the architecture; Phase H evolves it.

### Question 16
**Correct Answer:** b) A gap is a difference between the Baseline Architecture and the Target Architecture; a requirement is a formal, implementable statement about what must be built or changed
**Explanation:** Gap Analysis identifies three types of differences: items in the target that are not in the baseline (new capabilities needed), items in the baseline that are not in the target (capabilities to be removed), and items that exist in both but need modification. These gaps are architectural observations. Requirements are the formal, actionable statements that emerge from analyzing these gaps (and other sources like stakeholder concerns). A gap says "we don't have X"; a requirement says "the implementation must deliver X with these specific characteristics."

### Question 17
**Correct Answer:** b) The Reference Library contains reference architectures, templates, patterns, and other reference material for guidance; the Standards Information Base captures the specific standards with which new architectures must comply
**Explanation:** The Reference Library is a collection of guidance materials—reference architectures, architecture patterns, templates, best practices, and other material that architects can reference when developing architectures. The Standards Information Base (SIB) is a collection of specific standards—technology standards, regulatory standards, organizational standards—that define mandatory compliance requirements. The Reference Library guides ("consider using this pattern"); the SIB mandates ("you must comply with this standard").

### Question 18
**Correct Answer:** b) The Architecture Board is a permanent governance body providing oversight and decision-making authority over architecture; an Architecture Review is a periodic assessment activity that evaluates a specific project or architecture
**Explanation:** The Architecture Board is an organizational structure—a standing body with defined membership that provides continuous governance oversight, makes decisions, resolves disputes, and oversees architecture change management. An Architecture Review is an activity—a time-bound assessment that evaluates a specific project's architecture or compliance against the enterprise architecture. The Board may commission reviews, receive review results, and act on review findings, but they are distinct: one is a body, the other is an activity.

### Question 19
**Correct Answer:** b) A Logical component represents an implementation-independent definition of functionality or capability; a Physical component represents a specific, deployment-ready implementation with technology choices made
**Explanation:** This is a fundamental TOGAF distinction. Logical components describe what is needed without prescribing how—they define functionality, services, and relationships at a technology-neutral level. Physical components describe specific implementations—actual products, configurations, and deployments. For example, a "Logical Application Component: Customer Relationship Management" describes the CRM capability needed; a "Physical Application Component: Salesforce Sales Cloud on AWS" describes the specific implementation. This logical-to-physical mapping is a key architecture activity.

### Question 20
**Correct Answer:** a) The Architecture Vision is a high-level, aspirational view with first-cut architecture descriptions; the Architecture Definition Document is the comprehensive, detailed architecture specification developed progressively across Phases A through D
**Explanation:** The Architecture Vision (Phase A deliverable) provides the initial, high-level view of the target architecture—enough to gain stakeholder buy-in and approval for detailed architecture work. The Architecture Definition Document (ADD) is progressively elaborated across Phases A-D, growing from the initial vision into a comprehensive specification covering business, data, application, and technology architecture in detail. The Vision is the "pitch"; the ADD is the "blueprint."

### Question 21
**Correct Answer:** b) Stakeholder Management is the process of identifying, classifying, and managing stakeholder concerns throughout the ADM; Architecture Governance is the framework of practices and structures for controlling and guiding architecture activities
**Explanation:** Stakeholder Management is an ongoing ADM activity focused on people—identifying who the stakeholders are, understanding their concerns, and managing communication and engagement. Architecture Governance is an organizational discipline focused on control—establishing the structures, processes, and policies that ensure architecture activities are properly directed and architecture decisions are properly implemented. Stakeholder management informs governance; governance provides the structural authority for architecture decisions.

### Question 22
**Correct Answer:** b) A catalog is a list of items of a specific type; a matrix shows relationships between two or more entities; a diagram provides a graphical representation of architecture
**Explanation:** TOGAF defines three core artifact types: Catalogs are structured lists of architectural entities (e.g., Technology Standards Catalog lists approved technologies; Application Portfolio Catalog lists applications). Matrices show relationships between entities (e.g., Application-to-Function Matrix shows which applications support which business functions). Diagrams are graphical representations (e.g., Application Communication Diagram shows data flows between applications). Each type serves a different purpose in documenting and communicating architecture.

### Question 23
**Correct Answer:** b) The Preliminary Phase establishes the architecture capability, tailors the framework, and defines principles; Phase A applies the ADM to a specific architecture engagement
**Explanation:** The Preliminary Phase is an organizational setup activity—it prepares the enterprise to do architecture work by establishing the architecture function, tailoring TOGAF to the organization, and defining enterprise-wide principles. It is done once (or infrequently) and establishes the capability. Phase A (Architecture Vision) is the first phase of each ADM cycle—it scopes a specific architecture engagement, develops a vision for that particular effort, and obtains approval to proceed. The Preliminary Phase sets up the "factory"; Phase A starts each "production run."

### Question 24
**Correct Answer:** a) Compliance is a broader assessment of adherence to architecture specifications; Conformance is a specific compliance level within the TOGAF framework
**Explanation:** Architecture Compliance is the overall discipline and process of assessing how well implementations adhere to the enterprise architecture—it encompasses the entire compliance assessment framework with its six levels. Conformance is one specific level within that framework—it means the implementation includes all features of the architecture specification. So compliance is the umbrella concept; conformance is a specific measurement within it.

### Question 25
**Correct Answer:** b) The Architecture Metamodel defines the types of entities and their relationships; the Architecture Landscape contains the actual architecture content
**Explanation:** The Architecture Metamodel is a structural definition—it describes what types of entities exist in the architecture (organizations, functions, applications, data entities, etc.) and how they relate to each other. Think of it as the "schema." The Architecture Landscape is the actual content—the specific models, catalogs, and diagrams that describe the enterprise's real architecture assets. Think of it as the "data." The metamodel defines the structure; the landscape contains the instances.

### Question 26
**Correct Answer:** b) Consistent means some common features but no formal conformance; Non-Conformant means explicitly does not conform despite being in the same functional area
**Explanation:** "Consistent" is a mild positive—there is some alignment between the implementation and the specification, but it is informal and incomplete. There is no conflict but no formal compliance claim either. "Non-Conformant" is explicitly negative—the implementation is in the same functional area as the specification but definitively does not conform to it. The key distinction: Consistent implies benign non-alignment; Non-Conformant implies active conflict with the specification.

### Question 27
**Correct Answer:** b) A Transition Architecture represents a formally defined intermediate state between Baseline and Target; the Target Architecture is the final desired end-state
**Explanation:** In TOGAF, the Target Architecture represents the desired end-state—where the enterprise wants to be after all transformation is complete. Transition Architectures are formally defined intermediate states (plateaus) that the enterprise moves through on the way to the target. Each Transition Architecture must be independently stable—the enterprise could remain at any plateau if circumstances change. There may be multiple Transition Architectures between baseline and target, defined in Phase E and sequenced in Phase F.

### Question 28
**Correct Answer:** b) Business Scenarios discovers business requirements by describing business problems; Gap Analysis compares baseline and target to identify differences
**Explanation:** Business Scenarios is a requirements discovery technique—it creates narrative descriptions of business situations (who is involved, what they're trying to do, what problems exist, what the desired outcome is) to elicit and document architecture requirements. Gap Analysis is an architecture comparison technique—it systematically compares the baseline architecture to the target architecture to identify what's new, what's removed, and what needs modification. Business Scenarios is used primarily in Phase A; Gap Analysis is used in Phases B, C, and D.

### Question 29
**Correct Answer:** b) The Governance Log is a repository component that records decisions; the Architecture Compliance process is the active assessment activity
**Explanation:** The Governance Log is a storage mechanism within the Architecture Repository—it records governance decisions, compliance assessment results, dispensations, and other governance artifacts. It is the "memory" of governance. The Architecture Compliance process is an active, structured assessment—a set of activities performed to evaluate whether implementations conform to the architecture. It produces findings that are then stored in the Governance Log. The process generates findings; the log stores them.

### Question 30
**Correct Answer:** b) A Business Service is business-visible and may include manual activities; an IS Service is automated and provided by application components
**Explanation:** In the TOGAF Content Metamodel, a Business Service is a service that supports business capabilities and is visible to business stakeholders—it may involve manual processes, human interactions, and automated components. An IS Service (Information System Service) is specifically an automated service provided by application components—it represents the technology-mediated interface through which applications expose their functionality. A Business Service may be supported by one or more IS Services, but may also include non-automated activities.

### Question 31
**Correct Answer:** b) Enterprise Architecture Skills cover the ADM, frameworks, and governance; Technical IT Skills cover specific technology knowledge
**Explanation:** Enterprise Architecture Skills in the TOGAF Architecture Skills Framework focus on architecture-level capabilities: understanding and applying the ADM, working with architecture frameworks, conducting architecture governance, developing architecture principles, and thinking at the enterprise level. Technical IT Skills focus on technology implementation capabilities: software engineering, database management, network architecture, security technologies, and specific platform expertise. Enterprise architects need both, but these are distinct skill categories with different knowledge domains.

### Question 32
**Correct Answer:** b) The Architecture Board is a governance body; the Chief Architect is an individual leader
**Explanation:** The Architecture Board is a collective governance body—it provides oversight, makes decisions, resolves disputes, and represents diverse stakeholder interests. It operates through consensus and formal processes. The Chief Architect is an individual leadership role—responsible for leading architecture development, managing the architecture team, and making day-to-day architectural decisions. The Chief Architect typically presents recommendations to the Architecture Board for approval and implements the Board's decisions.

### Question 33
**Correct Answer:** b) Phase H manages changes to the established architecture; Requirements Management continuously manages requirements across all phases
**Explanation:** Phase H (Architecture Change Management) is specifically about managing changes to the architecture baseline—it provides a structured process for evaluating and incorporating changes driven by new technology, business changes, or lessons learned from implementation. Requirements Management is the central, continuous process that operates throughout ALL ADM phases—it ensures requirements are captured, maintained, and properly addressed as input to each phase. Phase H changes the architecture; Requirements Management ensures requirements are consistently managed across the entire ADM lifecycle.

### Question 34
**Correct Answer:** b) An Architecture Principle is a general rule guiding decisions; an Architecture Standard is a specific, enforceable specification
**Explanation:** Architecture Principles are high-level, enduring guidelines that express organizational values and inform decision-making (e.g., "Prefer open standards over proprietary solutions"). Architecture Standards are specific, measurable, enforceable specifications (e.g., "All web services must use REST over HTTPS with OAuth 2.0 authentication"). Principles guide the spirit of architecture decisions; standards specify the letter. Principles are relatively stable; standards may change as technology evolves. Standards should align with and implement principles.

### Question 35
**Correct Answer:** a) The Stakeholder Map Matrix maps stakeholders to their architectural concerns to identify which views are needed; the Power/Interest Grid classifies stakeholders by power and interest for engagement strategy
**Explanation:** The Stakeholder Map Matrix is a TOGAF-specific tool that creates a mapping between stakeholders and their architectural concerns—helping architects understand what each stakeholder cares about and which architectural views need to be created to address those concerns. The Power/Interest Grid (used in general stakeholder management) classifies stakeholders into four quadrants based on their level of organizational power and their degree of interest in the architecture initiative—determining whether to "manage closely," "keep satisfied," "keep informed," or "monitor."

### Question 36
**Correct Answer:** b) Architecture Capability defines organizational structures and processes; Architecture Landscape contains actual architectural content
**Explanation:** Architecture Capability in the Architecture Repository defines how the architecture function operates—the organizational structures, roles, skills, processes, and governance mechanisms that support architecture activities. It describes the "how we do architecture." Architecture Landscape contains the actual architecture content—the Strategic, Segment, and Capability level representations of the enterprise's architectural assets. It describes "what our architecture looks like." Capability is about the architecture organization; Landscape is about the architecture content.

### Question 37
**Correct Answer:** b) Implementation Factor Assessment evaluates factors influencing implementation sequencing; Business Transformation Readiness Assessment evaluates organizational readiness for change
**Explanation:** The Implementation Factor Assessment (Phase F, Migration Planning) evaluates specific factors that influence how implementation projects should be sequenced—costs, resource availability, risks, benefits, dependencies, and organizational politics. It is used to create the migration sequence. The Business Transformation Readiness Assessment (used in conjunction with Phase E/F) evaluates the organization's overall capacity for change—people readiness, process maturity, cultural factors, and change management capability. One is about project sequencing; the other is about organizational capacity for change.

### Question 38
**Correct Answer:** b) Phase G contracts govern implementation teams' conformance to target architecture; partner contracts govern collaborative architecture development
**Explanation:** TOGAF describes two contexts for Architecture Contracts. Phase G Architecture Contracts are between the architecture governance function and implementation/delivery teams—they ensure that what gets built conforms to what was designed. Architecture Contracts between development partners govern how multiple organizations collaboratively develop and share architecture work products—they define responsibilities, deliverables, and quality standards for architecture development collaboration. Both are formal agreements, but they govern different relationships and activities.

### Question 39
**Correct Answer:** b) The Implementation Factor Assessment identifies factors; the Deduction Matrix uses those factors to derive migration priorities
**Explanation:** These are complementary tools used in Phase F (Migration Planning). The Implementation Factor Assessment identifies and evaluates the factors that influence implementation—such as risks, costs, organizational impact, dependencies, and benefits for each candidate project. The Deduction Matrix then cross-references these assessed factors against the candidate migration projects to systematically derive the optimal implementation sequence. The Assessment provides the inputs; the Matrix provides the decision framework for using those inputs.

### Question 40
**Correct Answer:** b) Phase A encompasses all activities; the Architecture Vision document is one specific deliverable
**Explanation:** This is an important distinction often confused by candidates. "Architecture Vision" as a phase (Phase A) encompasses a full set of activities: establishing the project scope, identifying stakeholders, conducting Business Scenarios, developing the initial architecture description, performing business value assessment, and obtaining approval. "Architecture Vision" as a document is one specific deliverable produced by Phase A—it contains the high-level aspirational architecture description. Other Phase A deliverables include the Statement of Architecture Work and the refined Stakeholder Map. The phase is the process; the document is one output of that process.
