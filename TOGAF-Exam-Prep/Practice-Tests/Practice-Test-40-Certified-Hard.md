## Scenario 1: Cybersecurity Crisis Blocking Cloud Architecture

PacificEnergy Corporation is a major energy utility operating critical infrastructure including power generation plants, transmission grids, and distribution networks serving 12 million customers across the western United States. The company completed Phases A through D of the ADM 6 months ago. The approved Phase D Technology Architecture mandates migrating 70% of IT workloads from aging on-premises data centers to a multi-cloud environment (AWS and Azure), projected to save $45 million annually and enable advanced grid analytics capabilities essential for the company's renewable energy transition strategy.

Three weeks ago, a sophisticated cyberattack targeted PacificEnergy's Supervisory Control and Data Acquisition (SCADA) systems — the operational technology (OT) that controls power generation and grid operations. While the attack was contained before any operational impact, the incident exposed critical vulnerabilities in the boundary between IT and OT networks. The attack vector exploited a cloud-based monitoring tool that had been deployed as part of an early pilot program.

The board of directors has responded with a "freeze" directive: no new cloud deployments until the CISO certifies that cloud environments can be secured to the same standard as on-premises infrastructure. The CISO, Dr. Yuki Tanaka, estimates this certification process will take 12-18 months and require $30 million in security infrastructure investment. The CIO argues that the cloud migration is essential for the renewable energy strategy and that the 18-month delay could cost the company $120 million in delayed grid analytics benefits. The CEO is caught between the board's risk aversion and the strategic imperative.

Further complicating matters: the state Public Utilities Commission (PUC) has opened an investigation into the cyberattack and is considering new cybersecurity regulations for utilities. Two of PacificEnergy's cloud migration contracts have "use it or lose it" clauses worth $8 million that expire in 6 months. The Architecture Board has been asked to recommend a path forward that addresses security concerns without abandoning the strategic technology direction.

### Question 1
Based on the scenario above, how should the Architecture Board navigate this situation?

a) Revise the Phase D Technology Architecture to incorporate a comprehensive security architecture that addresses IT/OT convergence risks. Create a "Secure Cloud" architecture pattern that includes: network segmentation between cloud-hosted IT workloads and OT systems, zero-trust architecture for all cloud access, a cloud security operations center (SOC), and a phased migration that begins with non-OT workloads (corporate IT, analytics, customer systems) while keeping all OT-adjacent systems on-premises until the CISO's certification is complete. Present this to the board as a risk-calibrated approach that enables strategic progress on low-risk workloads while respecting the freeze directive for OT-related systems.

b) Support the board's freeze directive and recommend pausing all cloud migration until the CISO's certification is complete. The cyberattack demonstrated that the Technology Architecture underestimated security risks. Use the 12-18 month pause to: conduct a thorough Phase D revision incorporating security architecture, develop cloud security capabilities, address the PUC's regulatory concerns, and redesign the IT/OT boundary architecture. Negotiate with cloud vendors to extend contract timelines. Present this to the Architecture Board as a necessary investment in architectural integrity — a flawed architecture deployed quickly is worse than a sound architecture deployed later.

c) Recommend that the Architecture Board commission an independent security assessment of the proposed cloud architecture by a firm specializing in critical infrastructure cybersecurity. The assessment should determine: which workloads can be safely migrated to cloud with existing security controls, which workloads require enhanced security measures before migration, and which workloads should remain on-premises permanently due to OT proximity. Use this assessment to create a risk-tiered migration plan that satisfies both the security concerns and the strategic timeline. Present the independent assessment to the board as objective evidence supporting a calibrated approach.

d) Accept that the cloud migration strategy must be fundamentally reconsidered. Propose a revised Technology Architecture that uses edge computing and on-premises private cloud instead of public cloud for all operations-related systems. This addresses the security concerns while still modernizing the infrastructure. Migrate only corporate IT functions (HR, finance, email) to public cloud. The grid analytics capabilities can be delivered through on-premises analytics platforms that have been hardened for critical infrastructure environments. Present this as an architecture that prioritizes security without sacrificing modernization.

## Scenario 2: Architecture Board Demanding Solution Building Blocks in the Wrong Phase

Nexus Biotech is a pharmaceutical research company ($8 billion revenue) developing next-generation gene therapies. The enterprise architecture team has completed Phase A (Architecture Vision) and Phase B (Business Architecture) for a major laboratory information management modernization initiative. The Chief Architect, Dr. Sarah Kim, is now presenting Phase C outputs — the Information Systems Architecture — to the Architecture Board.

The Phase C deliverables include Application Architecture definitions using Architecture Building Blocks (ABBs): "Laboratory Data Management Service" (captures, validates, and stores experimental data), "Regulatory Submission Engine" (compiles and formats data for FDA/EMA submissions), "Clinical Trial Data Integration Service" (integrates data across 45 active clinical trials), and "Research Collaboration Platform" (enables secure data sharing between internal and external research partners). Each ABB defines functionality, interfaces, dependencies, and quality attributes without specifying vendor products or implementation technologies.

The Architecture Board, chaired by the CTO, has rejected the Phase C deliverables. The CTO states: "I can't make investment decisions based on abstract building blocks. I need to know: which vendor's LIMS (Laboratory Information Management System) we're buying, what database will store the clinical trial data, and whether we're using Veeva or a competitor for regulatory submissions. Give me Solution Building Blocks with vendor names, product versions, and cost estimates. I won't approve Phase C until I see specific products."

Dr. Kim knows that Solution Building Blocks (SBBs) — which specify vendor products and implementation details — are Phase D (Technology Architecture) and Phase E (Opportunities and Solutions) deliverables, not Phase C. However, the CTO has the authority to withhold Phase C approval, which blocks the entire ADM cycle. Three board members support the CTO (they want vendor specifics too), two support Dr. Kim (they understand the ADM sequence), and one is neutral. The modernization initiative has a 6-month window before the current LIMS vendor's license expires, creating time pressure.

### Question 2
Based on the scenario above, how should Dr. Kim handle the Architecture Board's demand?

a) Educate the Architecture Board on the TOGAF distinction between Architecture Building Blocks (ABBs) and Solution Building Blocks (SBBs) and why the sequence matters. Explain that ABBs define "what" the architecture needs without constraining "how" — this ensures that vendor selection in Phase D is driven by architecture requirements, not vendor marketing. Selecting vendors before the information systems architecture is complete risks choosing products that don't fully satisfy the requirements. Request that the Board approve Phase C as delivered and commit to fast-tracking Phase D to deliver SBBs within 6 weeks. Offer to present preliminary vendor landscape analysis alongside Phase C to address the Board's interest in vendor options.

b) Accept the Architecture Board's feedback and revise the Phase C deliverables to include Solution Building Blocks alongside the Architecture Building Blocks. For each ABB, provide a recommended SBB with vendor name, product version, licensing model, and estimated cost. TOGAF permits tailoring the ADM to organizational needs, and if the Architecture Board requires vendor specifics at this stage to make informed decisions, the architect should accommodate the governance body's requirements. Document this as a tailoring decision — combining Phase C and Phase D deliverables to match the Architecture Board's decision-making needs.

c) Propose a compromise: maintain the Phase C ABB-level deliverables as the approved architecture but create an accompanying "Vendor Mapping Analysis" document that maps each ABB to 2-3 potential vendor products with high-level comparison. This gives the Architecture Board the vendor context they want without formally making vendor selection part of Phase C. The Vendor Mapping is explicitly labeled as "preliminary" and "subject to Phase D evaluation." This approach maintains the ADM's logical sequence while pragmatically addressing the board's information needs. Request Phase C approval with the vendor mapping as a supplementary deliverable.

d) Escalate the impasse to the executive sponsor (CEO or equivalent). The CTO is conflating architecture definition with procurement, and the Architecture Board's demand for vendor-specific deliverables in Phase C could lead to premature vendor lock-in. The 6-month license expiration is creating artificial urgency that could compromise architecture quality. The executive sponsor should direct the Architecture Board to follow the approved governance framework and ADM process. If the CTO wants to begin vendor evaluation in parallel, that can be managed as a separate procurement activity outside the ADM.

## Scenario 3: Circular Dependencies Among Work Packages

OmniLogistics International is a $20 billion global logistics company operating warehouses, transportation fleets, and customs brokerage services in 50 countries. The enterprise architecture team has completed Phases A through D and is now in Phase E (Opportunities and Solutions), where they have identified 8 work packages to realize the target architecture for the company's digital supply chain transformation.

The work packages are: WP1 — Global Shipment Visibility Platform (real-time tracking), WP2 — Automated Customs Processing Engine, WP3 — Warehouse Management System modernization, WP4 — Transportation Management Optimization (AI-driven route planning), WP5 — Customer Self-Service Portal, WP6 — Enterprise Data Lake (unified supply chain analytics), WP7 — Partner Integration Gateway (connects with 500+ logistics partners), WP8 — Compliance and Audit Automation.

Analysis reveals circular dependencies: WP1 (Visibility) needs data from WP6 (Data Lake), but WP6 needs real-time event feeds from WP1. WP2 (Customs) needs partner connectivity from WP7 (Integration Gateway), but WP7 needs customs classification data from WP2 to route documents correctly. WP3 (Warehouse) needs route optimization from WP4 (Transportation) to schedule outbound shipments, but WP4 needs real-time warehouse inventory from WP3 to optimize loads. WP5 (Customer Portal) needs visibility data from WP1 and customs status from WP2, creating additional dependencies that feed back through the circular chains.

The program management team has declared the work package plan "unimplementable" because no work package can start without another being complete first. The program's $180 million budget has been approved and 500 staff have been allocated, but implementation cannot begin until the dependency deadlock is resolved. The CIO has given the architecture team 3 weeks to resolve the issue or she will "restructure the program without architecture involvement."

### Question 3
Based on the scenario above, how should the enterprise architect break the circular dependency deadlock?

a) Restructure the 8 work packages into 3 larger, integrated work packages that eliminate the circular dependencies by combining interdependent components. Work Package A: "Supply Chain Data Foundation" (combines WP1, WP6, and WP7 — the data platform, visibility, and integration components that are deeply interdependent). Work Package B: "Operations Modernization" (combines WP2, WP3, and WP4 — the operational systems that need each other). Work Package C: "Customer and Compliance" (combines WP5 and WP8 — the customer-facing and compliance systems that consume data from A and B). Implement in sequence: A first (9 months), then B (9 months), then C (6 months).

b) Break the circular dependencies by defining interface contracts (APIs, event schemas, data formats) for each work package before implementation begins. Create a 4-week "Architecture Sprint" that produces the interface specifications for all 8 work packages. Once interfaces are defined, each work package can develop against the contracted interfaces using stubs and simulators for dependencies that aren't yet built. All 8 work packages can then proceed in parallel with staggered deliveries. Use continuous integration testing to validate interface conformance as real implementations replace stubs. This approach leverages the ABB definitions from Phase C to create implementable interface contracts.

c) Identify the "foundation" services that break the most dependency cycles and implement them as a dedicated precursor phase. The Enterprise Data Lake (WP6) and Partner Integration Gateway (WP7) are the most referenced dependencies. Implement minimal viable versions of WP6 and WP7 as "Platform Foundation" in a 4-month sprint, providing basic data storage and partner connectivity. Then launch the remaining 6 work packages in parallel, each building on the foundation platform. This "foundation first" approach breaks the largest dependency cycles while enabling parallel execution of the remaining work packages.

d) Apply TOGAF's Transition Architecture technique to sequence the work packages across 3 transition states, each delivering working capability. Transition 1 (months 1-8): WP6 (Data Lake core) + WP7 (Integration Gateway core) + WP1 (Visibility MVP using simplified data feeds). Transition 2 (months 6-14, overlapping): WP3 (Warehouse with basic optimization) + WP4 (Transportation with basic warehouse data) + WP2 (Customs with basic partner connectivity). Transition 3 (months 12-20): WP5 (Customer Portal consuming Transition 1&2 outputs) + WP8 (Compliance using accumulated data). Each work package starts with simplified versions of its dependencies and evolves as predecessors mature. Define architectural "interfaces" at each transition boundary.

## Scenario 4: Enterprise Architecture for a Startup Culture

HyperScale AI is a 3-year-old artificial intelligence startup that has grown from 50 to 2,500 employees with $500 million in annual recurring revenue. The company's AI platform serves 5,000 enterprise customers and processes 50 petabytes of data daily. The company has raised $2 billion in venture capital and is preparing for an IPO in 12 months.

The board of directors has mandated the creation of an enterprise architecture function because due diligence for the IPO revealed alarming technical conditions: there are 4 different customer databases with no single source of truth, 3 separate billing systems, no documented technology architecture, security practices are inconsistent, and there is no disaster recovery plan for the core AI platform. The company's infrastructure grew organically as engineering teams built what they needed, when they needed it.

The challenge: HyperScale's engineering culture is aggressively anti-governance. The VP of Engineering, Jake Torres, has publicly stated: "Process kills innovation. We got to $500 million by moving fast and shipping code, not by filling out architecture review forms." His 800 engineers share this sentiment. The CEO values the engineering culture as a competitive advantage but recognizes that IPO readiness requires "adult supervision" of technology decisions. She has hired Maria Gonzalez, an experienced enterprise architect, as the company's first Chief Architect. Maria has a team of 3 architects and a mandate to "make us IPO-ready without killing the culture."

Maria faces specific challenges: the IPO window is 12 months (no time for multi-year architecture programs), the engineering teams will reject anything that looks like "traditional enterprise architecture," the 4 customer databases and 3 billing systems create real financial reporting risk that auditors will flag, and the lack of disaster recovery for a platform processing 50 PB daily is an existential business risk that potential investors will scrutinize.

### Question 4
Based on the scenario above, how should Maria start building enterprise architecture at HyperScale?

a) Start with a tailored Preliminary Phase that establishes minimal, high-impact architecture governance. Define 5-7 architecture principles collaboratively with engineering leadership (not imposed from above) — focused on principles they already implicitly follow (e.g., "automate everything," "design for scale") plus principles they need (e.g., "single source of truth for financial data," "resilience by design"). Skip the formal Architecture Board — instead, embed architecture review into the existing engineering review processes (code reviews, design docs, sprint planning). Launch two focused ADM micro-cycles: one for Customer Data Consolidation (directly addressing the IPO audit risk) and one for Disaster Recovery Architecture. Target 6-month delivery for both.

b) Conduct a rapid Architecture Maturity Assessment and present the findings to the board, CEO, and VP of Engineering. Show the specific IPO risks created by the current state: auditor concerns about financial data integrity (4 customer databases), investor concerns about platform resilience (no DR), and security liability (inconsistent practices). Use this risk-based narrative to build organizational buy-in for architecture governance. Then establish a lightweight governance framework with the engineering leadership's input, prioritizing the areas that directly impact IPO readiness. Create an architecture roadmap with three streams: Data Consolidation (3 months), Platform Resilience (4 months), and Security Baseline (6 months).

c) Accept that a traditional TOGAF ADM approach won't work in this culture and adapt the methodology. Rename everything: "Architecture Review Board" becomes "Tech Strategy Council," "Architecture Building Blocks" become "Platform Capabilities," "Architecture Compliance Review" becomes "Engineering Quality Check." Use terminology that resonates with the engineering culture. Apply the ADM concepts without the TOGAF labels: Preliminary Phase becomes "Strategy Alignment," Phase B becomes "Business Capability Mapping," Phase D becomes "Platform Architecture." Create a visual architecture roadmap that looks like a product roadmap (familiar to engineers) rather than a traditional architecture document.

d) Focus entirely on the two existential risks: Customer Data Consolidation and Disaster Recovery. Skip all architecture framework setup, governance, and principles. Directly build a unified customer data platform by selecting one of the 4 databases as the master and migrating the others. Simultaneously, design and implement a disaster recovery plan for the AI platform. These two deliverables directly address IPO audit requirements. Once the IPO is complete and the company has more stability, establish the formal enterprise architecture practice. The 12-month window is too short for both architecture practice development and risk mitigation — prioritize the latter.

## Scenario 5: Conflicting Architecture Contracts

Meridian Health Systems is a $20 billion healthcare provider operating 30 hospitals and 500 clinics. Two architecture contracts are now in direct conflict, and the Architecture Board must adjudicate.

**Contract A** was established 18 months ago for the "Clinical Operations Modernization" program ($120 million, 3-year timeline). This architecture contract specifies: all clinical systems will migrate to a unified Epic EHR (Electronic Health Records) platform, all data integration will use HL7 FHIR APIs, and all clinical workflow automation will be built on the Epic platform's native workflow engine. The contract was approved by the Architecture Board and is 40% complete — $48 million has been spent, Epic implementation is underway in 12 hospitals, and 200 staff have been trained.

**Contract B** was established 3 months ago for the "AI-Driven Precision Medicine" strategic initiative ($80 million, 2-year timeline). This initiative is the CEO's top priority, driven by a partnership with a major academic medical center. Contract B's architecture specifies: clinical data will be accessed through a vendor-neutral data aggregation layer (not Epic-specific APIs), AI models will be deployed on a separate analytics platform that ingests data from multiple sources (not just Epic), and clinical decision support will be delivered through a standalone AI interface that operates alongside (not within) the Epic workflow engine.

The conflicts are clear: Contract A mandates Epic-centric data integration while Contract B requires vendor-neutral data access. Contract A mandates Epic-native workflow automation while Contract B requires independent AI-driven clinical decision support. If both contracts are honored as written, the organization will build two parallel, incompatible approaches to clinical data management and workflow automation.

The Chief Medical Officer (CMO) sponsors Contract B and insists that clinical AI cannot be locked into a single EHR vendor. The CIO sponsors Contract A and argues that architectural consistency through a unified platform is essential for operational reliability. The CEO has told the Architecture Board: "I need both initiatives to succeed. Find a way."

### Question 5
Based on the scenario above, how should the Architecture Board adjudicate between the two conflicting architecture contracts?

a) Modify Contract A to accommodate Contract B's requirements. Add a vendor-neutral data abstraction layer on top of Epic that provides the FHIR-based data access that both contracts need. Allow the AI platform to operate alongside Epic's workflow engine, with integration points for clinical decision support. This preserves the Epic platform investment (already 40% complete) while enabling the vendor-neutral data access and independent AI capabilities that Contract B requires. The modification acknowledges that the original Contract A was too restrictive in mandating Epic-only data access and workflow. Update the target architecture to reflect this hybrid approach.

b) Prioritize Contract B and modify Contract A accordingly. The AI-Driven Precision Medicine initiative is the CEO's top priority and represents the future of clinical care. Revise Contract A to adopt vendor-neutral data integration standards (not Epic-specific) and allow non-Epic clinical decision support tools. The Epic platform remains the primary EHR, but the architecture becomes Epic-agnostic for data access and workflow automation. This requires rework on the 12 hospitals already implementing Epic-specific integrations (estimated $15 million additional cost) but creates an architecture that supports both current operations and future AI capabilities.

c) Initiate a targeted ADM iteration — specifically Phase C (Information Systems Architecture) and Phase D (Technology Architecture) — to develop a unified information systems architecture that satisfies both contracts' objectives. The Architecture Board should recognize that both contracts represent legitimate architectural needs that were not reconciled because they were developed at different times. The ADM iteration should produce: a unified data architecture that serves both Epic operations and AI analytics, an application architecture that defines how Epic and AI platforms coexist, and revised architecture contracts for both programs that reflect the reconciled architecture. Suspend new implementation work on conflicting elements until the reconciled architecture is approved.

d) Maintain both contracts as-is and manage the conflicts through an integration architecture. Build a middleware layer that translates between Epic-centric and vendor-neutral data formats, allowing both approaches to coexist. The integration architecture serves as a Transition Architecture — it's not the ideal long-term solution, but it enables both programs to proceed without rework. Plan to reconcile the architectures in a future ADM cycle once both programs have delivered their initial capabilities and the organization has more data on how clinical AI actually integrates with EHR workflows. Document this as a planned technical debt with a committed resolution timeline.

## Scenario 6: Overcoming Political Resistance to Process Consolidation

FederalBank Corporation is a $100 billion banking group formed through 4 acquisitions over 10 years. The enterprise architecture team has completed Phase B (Business Architecture), revealing that 40% of the bank's 2,800 business processes are redundant — the same process exists in different forms across the legacy banking entities. For example, there are 5 different mortgage origination processes, 4 different account opening processes, 3 different compliance reporting processes, and 6 different customer complaint handling processes.

The redundancy costs an estimated $200 million annually in duplicated staff, redundant systems, inconsistent customer experiences, and compliance risk (different processes produce different compliance outcomes). The target Business Architecture proposes consolidating to a single, optimized version of each process, which would reduce the 2,800 processes to approximately 1,700 and save $130 million annually.

The political resistance is intense. Each legacy banking entity has a divisional president who controls their division's processes and staff. Process consolidation means: some divisions lose their unique processes (perceived as loss of identity and control), staff reductions of approximately 800 FTE across the organization, and one division's process becomes the "winner" while others' are "losers." The divisional presidents are powerful — they sit on the bank's Executive Committee and each threatens to escalate to the board of directors if their processes are targeted.

The CEO supports the consolidation in principle but has told the Chief Architect: "I need you to show me HOW to do this without a palace revolt. I can't force 4 division presidents to give up their processes — I need them to want to." The Architecture Board has approved the target Business Architecture, but implementation requires division-level cooperation that the Architecture Board cannot mandate.

### Question 6
Based on the scenario above, what strategy should the Chief Architect employ to overcome political resistance to process consolidation?

a) Create a "Best Practice Selection" process that gives each division fair representation. For each redundant process group (e.g., mortgage origination), form a cross-divisional working group that evaluates all versions against objective criteria: customer satisfaction scores, processing time, error rates, compliance audit results, and cost per transaction. Select the best-performing version regardless of which division owns it and use it as the basis for the consolidated process. Present this to the divisional presidents as an evidence-based, merit-based approach where the best process wins. Each division will "win" some processes and "lose" others, creating a perceived balance.

b) Reframe the consolidation as a "process innovation" initiative rather than a cost-cutting exercise. Instead of selecting one division's existing process as the "winner," design entirely new processes that incorporate the best elements from all divisions' versions. Engage process owners from all divisions as co-creators of the new processes, giving them ownership and recognition. Sequence the consolidation by business function (start with back-office processes that are least politically sensitive, then move to customer-facing processes). Create a shared services model where consolidated processes are owned by a cross-divisional team, not by any single division.

c) Build the case for consolidation through customer experience data. Commission a customer journey analysis that shows how the redundant processes create inconsistent experiences — a customer opening an account at Division A gets a different experience than at Division B, and customers who interact with multiple divisions have fragmented experiences. Present this data to the divisional presidents to demonstrate that the redundancy hurts customers, not just costs money. Then propose consolidation as a customer experience improvement, with a governance model where each phase of consolidation is validated by customer satisfaction data. This shifts the narrative from "lose your process" to "improve the customer."

d) Propose a federated process model: establish enterprise process standards (mandatory outcomes, compliance requirements, customer experience metrics) but allow each division to implement these standards in their own way. Instead of forcing process consolidation, ensure all divisions' processes achieve the same results. Use TOGAF's Architecture Compliance Review process to assess whether each division's process meets the enterprise standard. Over time, divisions will naturally converge as they optimize to meet the same standards, and the convergence can be accelerated through shared technology platforms. This avoids the political confrontation while driving gradual standardization.

## Scenario 7: Making the Architecture Definition Document Consumable

Quantum Defense Systems is a defense contractor ($25 billion revenue) that has just completed a comprehensive ADM cycle for its enterprise digital transformation. The architecture team has produced an Architecture Definition Document (ADD) that is thorough, technically precise, and fully compliant with TOGAF's content framework. The problem: it is 312 pages long, covers all four architecture domains in exhaustive detail, includes 85 architecture diagrams using ArchiMate notation, and references 340 architecture requirements.

The document has been distributed to 45 stakeholders for review. After 4 weeks, the response rate is 8% — only 4 stakeholders have provided feedback (all of them architects). The CIO's office has informally communicated that "nobody has time to read a 300-page document." The program manager for the $500 million digital transformation has stated that "my team doesn't understand ArchiMate, and the document doesn't tell us what to do next week." Business stakeholders have ignored the document entirely. The Head of Business Development says: "I asked for an architecture that would help us win more contracts, and you gave me a telephone book."

The Architecture Board has approved the ADD's technical content but has directed the architecture team to "make the architecture accessible to non-architects within 30 days." The team of 8 architects is frustrated — they believe the document is comprehensive and technically correct, and that stakeholders should invest the effort to read it. The Chief Architect recognizes this as a communication failure that threatens the architecture's adoption and the architecture team's credibility.

### Question 7
Based on the scenario above, how should the architecture team improve architecture communication?

a) Create a suite of stakeholder-tailored architecture views as defined by TOGAF's stakeholder management approach. For each stakeholder group, produce a targeted communication artifact: Executive Summary (3 pages) for C-suite — strategic intent, investment required, expected business outcomes. Business View (10 pages) for business leaders — business capability changes, process impacts, timeline. Implementation Roadmap (15 pages) for program managers — work packages, dependencies, milestones, resource requirements. Technical Architecture Guide (30 pages) for engineers — technology standards, integration patterns, design constraints. Retain the full 312-page ADD as the authoritative reference. Replace ArchiMate diagrams with simplified visuals for non-technical audiences.

b) Replace the monolithic ADD with a digital architecture portal that provides interactive, searchable architecture content. Convert the 312 pages into a wiki-style knowledge base where each architecture element (building block, requirement, principle, standard) is a discrete, linked page. Create role-based views that filter content by stakeholder type. Add a "What does this mean for me?" section to each page explaining the impact on specific roles. Include interactive architecture diagrams that allow users to explore details on demand. This approach makes the architecture discoverable rather than consumable — stakeholders find what they need rather than reading what the architects think they should read.

c) Schedule a series of architecture workshops — one for each major stakeholder group — where the architecture team presents the relevant portions of the ADD in interactive, discussion-based sessions. Use these workshops to gather feedback, answer questions, and build shared understanding. Create concise (5-10 page) "Architecture Decision Briefings" for each major architectural decision, explaining the decision context, alternatives considered, the chosen approach, and the implications for each stakeholder group. Follow each workshop with a one-page summary of key points discussed and decisions confirmed.

d) Admit that the 312-page ADD was a mistake and start over. Create a "Minimum Viable Architecture Document" of no more than 40 pages that covers only the architecture decisions that stakeholders need to act on — not the entire architecture landscape. Focus on: the 15 most impactful architecture decisions, the implementation roadmap, the technology standards that constrain implementation, and the governance framework for decision-making during delivery. Archive the 312-page ADD as reference material. This approach recognizes that architecture documentation is a communication tool, not a knowledge repository.

## Scenario 8: Tailoring TOGAF for a Resource-Constrained Non-Profit

Global Health Alliance (GHA) is an international non-profit organization with $800 million in annual revenue from government grants, private donations, and foundation funding. GHA operates in 45 countries, providing healthcare services, disease prevention programs, and health worker training. The organization has a central IT team of 40 people and an annual IT budget of $12 million (1.5% of revenue, compared to the 5-8% typical of similar organizations).

GHA's technology landscape is fragmented: different country offices use different systems for patient records, supply chain, human resources, and financial management. There are 23 different patient record systems, 15 different supply chain tools, and 8 different financial systems. Data cannot be aggregated across countries for program evaluation or donor reporting without extensive manual effort. A recent audit revealed that GHA spends 60% of its IT budget ($7.2 million) on maintaining this fragmented landscape and only 40% ($4.8 million) on new capabilities.

The CEO recognizes that technology fragmentation undermines GHA's mission effectiveness and donor confidence. She has asked the IT Director, Samuel Oduya (who has general IT management experience but no formal architecture training), to "create an enterprise architecture that unifies our technology." Samuel has researched TOGAF and believes it's the right approach, but he faces severe constraints: no budget for external architecture consultants, no staff with TOGAF training, no dedicated architecture team (architects must come from existing IT staff who have day jobs), no organizational experience with governance processes, and extreme pressure from field offices who resist any change to their systems because "lives depend on our systems working").

The board has committed to a 3-year technology consolidation but expects the first tangible results within 12 months. Donor reporting and program evaluation data — currently requiring weeks of manual compilation — is the highest-priority pain point.

### Question 8
Based on the scenario above, how should Samuel tailor the ADM for GHA's resource-constrained environment?

a) Implement a minimal ADM that focuses only on the phases that directly produce results. Skip the Preliminary Phase formalities — instead, establish 3-4 architecture principles through a single leadership workshop (e.g., "common platforms where possible, local flexibility where necessary," "donor reporting requirements drive data architecture"). Go directly to Phase A: create an Architecture Vision focused solely on unified donor reporting and program evaluation. Move quickly through Phase B (Business Architecture for reporting workflows) and Phase C (Data Architecture for cross-country data integration). Skip Phase D (use existing technology in country offices) and go to Phase E/F to plan the data integration migration. Assign 2-3 IT staff as part-time architects for this focused initiative. Deliver a unified reporting capability within 12 months.

b) Invest the first 6 months in building architecture capability before attempting any ADM cycle. Send 2-3 IT staff to TOGAF certification training. Establish a basic Architecture Repository using free/open-source tools. Create architecture principles through stakeholder workshops in 3-4 representative country offices. Form an Architecture Board with IT, program, and finance representation. Only after these foundational elements are in place (month 7), begin a formal ADM cycle starting from Phase A. This ensures the architecture practice is sustainable even though it delays tangible results.

c) Adopt an "Architecture Sprints" approach that uses the ADM iteratively on small, high-value initiatives. Sprint 1 (months 1-4): Focus on donor reporting data architecture — define a common data model for program outcomes across all 45 countries, select one lightweight data aggregation platform, and deploy to 5 pilot countries. Sprint 2 (months 5-8): Extend data integration to 20 countries based on Sprint 1 learnings, and begin supply chain data standardization. Sprint 3 (months 9-12): Complete data integration to all 45 countries and deliver the unified reporting dashboard. Each sprint uses a compressed ADM cycle (focused Phase A, B, and C) tailored to the sprint's scope. Governance is lightweight — architecture decisions are made by a steering committee of 5 (IT Director, Finance Director, 2 Program Directors, 1 Country Director).

d) Rather than attempting to implement TOGAF formally, use TOGAF as a reference framework. Extract the concepts most relevant to GHA's situation: Architecture Principles (to guide decisions), Enterprise Continuum (to classify reusable components), and Transition Architectures (to plan phased migration). Don't follow the ADM phases formally — instead, focus on the highest-value outcome: create a common data model for program data that all country offices can adopt. Select one open-source platform (like DHIS2, widely used in global health) as the standard for health data. Mandate adoption for all new implementations while allowing existing systems to continue until their natural replacement cycles. This pragmatic approach uses TOGAF concepts without TOGAF process overhead.

### Scenario 3 - Question 3
**Scoring:**
- **Answer d) = 5 points (Best):** This approach best applies TOGAF's Transition Architecture concept to the work package dependency problem. Transition Architectures are specifically designed to define intermediate states between baseline and target that deliver working capability. By creating three transition states with overlapping timelines, each work package can start with simplified versions of its dependencies (MVP data feeds, basic connectivity, basic optimization) and mature as predecessor capabilities are delivered. This is the essence of iterative architecture — you don't need the full target capability from day one; you need enough to make progress. The overlap between transitions enables parallel execution while respecting dependencies. The "architectural interfaces at each transition boundary" ensure that each transition's outputs are consumable by subsequent transitions. This approach maintains all 8 work packages as defined while resolving the dependency deadlock through sequencing and interface definition.
- **Answer b) = 3 points (Second):** The interface contract approach is architecturally sound — defining APIs and event schemas upfront enables parallel development with stubs. This is a well-established software engineering practice (contract-first design) that leverages Phase C's ABB definitions. The 4-week Architecture Sprint is practical and time-boxed. However, the approach assumes that the interfaces can be fully defined before implementation begins, which may not be true for complex integration patterns (some interface details emerge during development). The risk of stub-based development is that the stubs may not accurately represent the real system's behavior, leading to integration failures when real services replace stubs. The approach works well for loose dependencies but may struggle with the tightly coupled circular dependencies described.
- **Answer c) = 1 point (Third):** The "foundation first" approach correctly identifies WP6 and WP7 as the most-referenced dependencies. Building them first does break the largest dependency cycles. However, a 4-month "minimal viable version" of the Enterprise Data Lake and Partner Integration Gateway may still leave significant dependency gaps for the remaining 6 work packages. The approach also creates a sequential bottleneck — if the foundation deliverables are delayed, all 6 remaining work packages are blocked. The approach is simpler than options b and d but carries more schedule risk due to the sequential dependency.

The 24-month sequential timeline (9+9+6 months) is significantly longer than a parallel approach would deliver. Combining deeply interdependent components into larger packages increases each package's complexity, risk, and management difficulty. A $60 million "Supply Chain Data Foundation" package combining WP1, WP6, and WP7 is too large for effective management. The approach also loses the 8 work packages' clean alignment to business capabilities, making stakeholder engagement and benefit tracking more difficult. TOGAF favors manageable, well-scoped work packages over monolithic programs.
