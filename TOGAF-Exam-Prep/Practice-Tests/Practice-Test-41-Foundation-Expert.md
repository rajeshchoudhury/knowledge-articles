## Answer Key

### Question 1
**Correct Answer:** a) Any collection of organizations that has a common set of goals and/or a single bottom line  
**Topic:** TOGAF Core Definitions  
**Explanation:** The TOGAF Standard provides a precise definition of "Enterprise" as any collection of organizations that has a common set of goals and/or a single bottom line. This is intentionally broad to encompass government agencies, company divisions, or entire corporations. The key elements are: it's a "collection of organizations" (not just a single entity), with "common goals" or a "single bottom line." Option (d) adds concepts not in the precise definition. Options (b) and (c) describe related but different concepts.

### Question 2
**Correct Answer:** b) The fundamental concepts or properties of a system in its environment embodied in its elements, relationships, and in the principles of its design and evolution  
**Topic:** Core Definitions - Architecture  
**Explanation:** TOGAF aligns its definition of architecture with ISO/IEC/IEEE 42010:2011, which defines architecture as "the fundamental concepts or properties of a system in its environment embodied in its elements, relationships, and in the principles of its design and evolution." Option (a) is a commonly seen paraphrase but is not the formal ISO-aligned definition. Option (c) describes an architectural description, not architecture itself. Option (d) is from an older definition not used in the current standard.

### Question 3
**Correct Answer:** c) Irrelevant, Conformant, Compliant, Consistent, Non-Conformant, Not-Assessed  
**Topic:** Architecture Compliance  
**Explanation:** The TOGAF Standard specifies six compliance assessment levels. The correct order from most to least conformant is: Irrelevant (architecture is not applicable), Conformant (fully implements all standards), Compliant (implements the spirit if not the letter), Consistent (has no conflicts but is not fully conformant), Non-Conformant (conflicts exist), and Not-Assessed (not yet reviewed). Many candidates confuse the order of Conformant and Compliant, or place Irrelevant incorrectly. Irrelevant is listed first because when an architecture domain doesn't apply, it is not a compliance issue.

### Question 4
**Correct Answer:** b) Organization Unit, Actor, Role, Business Service, Information System Service, Data Entity, Application Component, Technology Component  
**Topic:** Content Metamodel  
**Explanation:** The TOGAF Content Metamodel defines core entities that form the foundation of architecture content. The complete list of core entities includes Organization Unit, Actor, Role, Business Service, Information System Service, Data Entity, Application Component, and Technology Component. Option (a) includes Platform Service which is an extension entity, not core. Option (c) splits Application Component into Logical and Physical which are extension-level distinctions. Option (d) omits Actor and Role which are core entities.

### Question 5
**Correct Answer:** a) To develop a high-level aspirational vision of the capabilities and business value to be delivered  
**Topic:** ADM Phase A Objectives  
**Explanation:** The stated objective of Phase A is to develop a high-level aspirational vision of the capabilities and business value to be delivered as a result of the proposed enterprise architecture. While obtaining management commitment (option b) is an important activity within Phase A, it is not the primary stated objective. Options (c) and (d) describe activities that occur within Phase A but do not capture the exact stated objective. The key word in the correct answer is "aspirational" which distinguishes it from detailed architecture work in later phases.

### Question 6
**Correct Answer:** c) Architecture Vision approved in Phase A  
**Topic:** ADM Phase Inputs and Outputs  
**Explanation:** The Architecture Vision document, approved as an output of Phase A, becomes a specific input to Phase B: Business Architecture. This is a straightforward inter-phase dependency. Architecture Principles (option a) are inputs to BOTH Phase A and Phase B. The Request for Architecture Work (option b) is an input to Phase A, not Phase B. The Enterprise Continuum (option d) is a reference resource available to all phases, not a specific phase input. The key distinction is understanding which artifacts flow specifically from one phase to the next.

### Question 7
**Correct Answer:** c) 6 components: Architecture Metamodel, Architecture Capability, Architecture Landscape, Standards Information Base, Reference Library, Governance Log  
**Topic:** Architecture Repository  
**Explanation:** The TOGAF Architecture Repository contains exactly six major components: (1) Architecture Metamodel, which describes the organizationally tailored application of an architecture framework; (2) Architecture Capability, which defines the parameters, structures, and processes supporting architecture governance; (3) Architecture Landscape, which presents architectural views of assets in use; (4) Standards Information Base (SIB), which captures standards with which architectures must comply; (5) Reference Library, which provides guidelines, templates, and patterns; and (6) Governance Log, which records governance activity. Option (d) adds a seventh component not separately identified. Option (b) omits the Governance Log.

### Question 8
**Correct Answer:** b) Platform Service

**Explanation:** In the TOGAF Content Metamodel, a Platform Service represents a unit of functionality that is meaningful to interoperability and communication between architecture descriptions and is provided by a technology platform. It is the technical service that supports application components. Building Block (option a) is a more general concept. Information System Service (option c) describes an automated service provided by an application. Logical Application Component (option d) is an encapsulation of application functionality that is independent of implementation.

### Question 9

While the Architecture Board interacts with stakeholders, it itself is a governance mechanism, not a stakeholder category. This distinction is crucial because governance bodies serve stakeholder interests but are not themselves classified as stakeholder categories in the TOGAF taxonomy.

### Question 31

**Explanation:** Business Scenarios is a specific technique recommended by the TOGAF Standard for use in Phase B: Business Architecture. Business Scenarios help to identify and understand business needs by describing a business process, the business and technology environment, the people and computing components, and the desired outcome. ### Question 33
**Correct Answer:** b) Developing the detailed architecture for each domain  
**Topic:** Architecture Board  
**Explanation:** The Architecture Board is a governance body responsible for oversight, not for developing detailed architecture. Its stated responsibilities include providing the basis for decision-making regarding changes, ensuring compliance with standards, establishing targets for re-use, and other governance functions. Developing detailed architecture (option b) is the responsibility of the architecture team and individual enterprise architects, not the Architecture Board. The Board reviews, approves, and governs architecture work—it does not perform the detailed architecture development itself. This is a critical governance distinction.

### Question 34

"Stakeholder Constraints" (option d) is not a named category in the standard.
