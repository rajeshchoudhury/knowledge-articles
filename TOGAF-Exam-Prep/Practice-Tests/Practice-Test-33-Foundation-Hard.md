## Answer Key

### Question 1
**Correct Answer:** c) Architecture Compliance Matrix  
**Difficulty:** Expert  
**Topic:** Architecture Repository Components  
**Explanation:** The standard components of the TOGAF Architecture Repository include: Architecture Metamodel, Architecture Capability, Architecture Landscape, Standards Information Base (SIB), Reference Library, and Governance Log. An "Architecture Compliance Matrix" is not a standard repository component—compliance assessments are tracked within the Governance Log, and compliance review is a process rather than a stored repository component. This question tests precise knowledge of the repository's defined structure.

---

### Question 2
**Correct Answer:** a) A high-level, long-term view providing an organizing framework for operational and change activity, typically spanning 3-10 years  
**Difficulty:** Expert  
**Topic:** Architecture Landscape - Strategic Architecture  
**Explanation:** A Strategic Architecture in the Architecture Landscape provides a high-level, long-term view of the enterprise (typically 3-10 years), establishing an organizing framework that guides operational and change activities. It is not a detailed implementation plan (option b)—that would be closer to a Capability Architecture. It is not a mid-level view of specific areas (option c)—that describes Segment Architecture. And it is not a comprehensive detailed architecture (option d)—it deliberately maintains a high level of abstraction to provide overarching direction.

---

### Question 3
**Correct Answer:** b) To describe a formal structure for the types of architecture content, their relationships, and constraints — enabling consistency and rigor in architecture deliverables  
**Difficulty:** Expert  
**Topic:** Architecture Metamodel  
**Explanation:** The Architecture Metamodel defines the formal structure of architecture content—specifying what entity types exist (e.g., Business Service, Application Component, Data Entity), how they relate to each other, and what constraints apply. This ensures that architecture content across the repository is structurally consistent and rigorous. It does not define technology standards (option a—that's the SIB), provide governance escalation paths (option c—that's governance framework), or store historical versions (option d—that's repository versioning).

---

### Question 4
**Correct Answer:** a) Strategic Architecture  
**Difficulty:** Expert  
**Topic:** Architecture Landscape Partitioning  
**Explanation:** The Strategic Architecture partition is most concerned with providing a summary view of the enterprise, making it ideal for communication with executive stakeholders. It provides the high-level, long-term perspective that executives need to understand the enterprise's architectural direction without getting into implementation details. Segment Architecture (option b) is too focused on specific business areas. Capability Architecture (option c) is too detailed for executive audiences. "Tactical Architecture" (option d) is not a standard TOGAF Architecture Landscape partition name.

---

### Question 5
**Correct Answer:** b) Standards covering all architecture domains including business, data, application, and technology standards, along with their lifecycle status  
**Difficulty:** Expert  
**Topic:** Standards Information Base Content  
**Explanation:** The Standards Information Base contains standards across all architecture domains—not just technology. It includes business standards, data standards, application standards, and technology standards, each with lifecycle status information (e.g., Trial, Active, Deprecated, Obsolete). It is not limited to technology products and vendors (option a), not restricted to external regulations (option c), and architecture principles are separate from standards even though they guide standard selection (option d).

---

### Question 6
**Correct Answer:** b) Reference materials, templates, patterns, and other reusable assets that provide guidance and starting points for architecture development  
**Difficulty:** Expert  
**Topic:** Reference Library Content  
**Explanation:** The Reference Library stores a broad range of reusable reference materials including reference models, architecture patterns, templates, guidelines, and other assets that architects can leverage during architecture development. It is not limited to only TOGAF's own reference models (option a)—it includes any relevant reference material. It is not restricted to industry-specific architectures only (option c). And it stores static reference content, not real-time performance metrics (option d).

---

### Question 7

**Difficulty:** Expert  
**Topic:** Governance Log  
**Explanation:** The Governance Log maintains a comprehensive record of governance activities including architecture decision logs, compliance review results, capability assessments, architecture contracts, dispensation records, and other governance-relevant documentation. It provides an auditable trail of governance actions. It is not limited to compliance reviews only (option a), does not contain personnel evaluations (option c), and does not store financial audit results (option d)—those belong in financial management systems.

It operates at a moderate level between the broad Strategic Architecture and the specific Capability Architecture. It is not enterprise-wide (option a—that's Strategic), not a single component (option c—too narrow), and not focused on external ecosystem (option d—that extends beyond the typical Segment scope).

### Question 9

It does not replace the ADM (option a), is not populated only after completion (option c), and stores content from all phases, not just Phase A (option d).

It provides the "how we do architecture" framework that enables the Architecture Landscape to be properly developed and maintained. They are not the same component (option b), the Architecture Capability is about the architecture practice itself, not technology capabilities (option c), and the relationship is Capability governing Landscape, not the reverse (option d).

It is not solely about audit compliance (option a), is not about repository size (option c), and versioning is about change management, not access control (option d). Good versioning practices are fundamental to effective architecture governance.

### Question 13
**Correct Answer:** b) Standards in the SIB have lifecycle states such as Trial, Active, Deprecated, and Obsolete, reflecting their current applicability and usage status  
**Difficulty:** Expert  
**Topic:** Standards Lifecycle Management  
**Explanation:** Standards in the SIB have defined lifecycle states that reflect their current status. A standard may be in Trial (being evaluated), Active (approved for use), Deprecated (still supported but being phased out), or Obsolete (no longer supported). Standards are not permanent and unchanging (option a). They are updated throughout the architecture lifecycle, not just in the Preliminary Phase (option c). And the SIB contains both internal and external standards, not just regulatory ones (option d).

### Question 14
**Correct Answer:** c) Phase A: Architecture Vision contributes the initial scope, and Phases B-D populate the detailed landscape content  
**Difficulty:** Expert  
**Topic:** Repository Population Across ADM  
**Explanation:** The Architecture Landscape is initially scoped in Phase A when the Architecture Vision defines the breadth and depth of the architecture effort. The detailed landscape content—business, data, application, and technology architectures—is then progressively populated through Phases B, C, and D. Phase H manages changes to the landscape (option a), the Preliminary Phase establishes the repository structure but not the landscape content (option b), and Phase G governs implementation but doesn't primarily populate the landscape (option d).

### Question 15
**Correct Answer:** a) The Architecture Landscape contains the organization's actual current, transition, and target architectures while the Reference Library contains reusable reference materials, templates, and patterns  
**Difficulty:** Expert  
**Topic:** Architecture Landscape vs Reference Library  
**Explanation:** The Architecture Landscape stores the organization's actual architectures—the current (Baseline) state, planned Transition Architectures, and the Target Architecture. The Reference Library stores reusable reference materials that architects can draw upon during development—templates, patterns, reference models, and guidelines. The Landscape is "what our architecture is"; the Reference Library is "what resources we can use to build architecture." They are not limited by strategic/tactical scope (option b), do not contain the same content (option c), and both are populated through the architecture practice, not by external vendors (option d).

---

### Question 16
**Correct Answer:** b) That changes to architecture artifacts are controlled, tracked, and traceable, maintaining the integrity and consistency of the repository  
**Difficulty:** Expert  
**Topic:** Configuration Management  
**Explanation:** Configuration management ensures that all changes to architecture artifacts in the repository are properly controlled (following defined change management processes), tracked (recorded with metadata about who changed what and when), and traceable (enabling understanding of the evolution of each artifact). It is not about restricting edits to one person (option a), not about encryption (option c), and not about storage optimization (option d). Configuration management is essential for maintaining a reliable, trustworthy architecture repository.

### Question 17
**Correct Answer:** c) Governance Log  
**Difficulty:** Expert  
**Topic:** Governance Log Content  
**Explanation:** A formal architecture dispensation (an approved deviation from architecture standards) is a governance decision that would be recorded in the Governance Log. The Governance Log captures all governance-related records including decisions, dispensations, compliance review outcomes, and architecture contracts. The SIB (option a) stores standards, not governance decisions. The Reference Library (option b) stores reusable reference materials. The Architecture Metamodel (option d) defines the structure of content, not governance records.

### Question 18

**Difficulty:** Expert  
**Topic:** Enterprise Continuum and Repository Relationship  
**Explanation:** The Enterprise Continuum serves as a classification framework (taxonomy) for organizing the architecture and solution content stored in the Architecture Repository. It is not a physical partition (option a)—it is a virtual/logical classification scheme. The Architecture Repository has not replaced the Enterprise Continuum (option c)—they serve complementary roles. And they serve different purposes (option d)—the Enterprise Continuum classifies content; the Repository stores and manages it.

### Question 19
**Correct Answer:** b) They form a hierarchy where the Strategic Architecture provides the overarching context, Segment Architectures detail specific areas within that context, and Capability Architectures define deployable increments  
**Difficulty:** Expert  
**Topic:** Architecture Landscape Hierarchy  
**Explanation:** The three Architecture Landscape partitions form a natural hierarchy. Strategic Architecture provides the broadest, most long-term view. Segment Architectures elaborate specific areas (business units, functions, or cross-cutting concerns) within the strategic context. Capability Architectures then define the most detailed, deployable increments within segments. They are not independent (option a). They are not strictly sequential—work at different levels can overlap (option c). And the hierarchy flows from Strategic to Capability, not the reverse (option d).

---

### Question 20
**Correct Answer:** c) Individual project status reports and timelines  
**Difficulty:** Expert  
**Topic:** Standards Information Base Scope  
**Explanation:** Individual project status reports and timelines are project management artifacts, not standards, and would be least appropriately stored in the Standards Information Base. The SIB is for standards—rules, guidelines, and specifications that define approved approaches. Technology product standards (option a), business service level standards (option b), and application development standards (option d) are all legitimate standards that belong in the SIB. Project status information belongs in project management repositories and potentially the Governance Log.

### Question 21

**Topic:** Architecture Metamodel Relationships  
**Explanation:** In the TOGAF Content Metamodel, "A Business Service is realized by a Logical Application Component" is a valid entity-relationship. Business Services represent what the business provides, and they can be realized (implemented) by Logical Application Components in the application architecture. A Data Entity being governed by a Technology Component (option b) is not a standard metamodel relationship. Platform Services don't manage Business Functions (option c). And Architecture Principles don't deploy Application Components (option d)—these are not valid metamodel relationships.

Versioning ensures historical content is preserved (option a is wrong about archiving outside). Content from all phases is retained, not just the most recent (option c). And the repository is designed to grow and evolve with each ADM cycle (option d is wrong about fixed content).
